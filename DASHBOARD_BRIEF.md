## The Bot — What Already Exists (Read This Carefully)

The bot (`grog-bot/index.js`) is a Node.js Discord bot using discord.js v14. Here is exactly what it does:

### Application Flow
1. User runs `/apply` in Discord
2. Bot queries Supabase for any existing applications from that Discord user, annotates a category select menu with status descriptions (pending / approved / reapply date)
3. User selects a category: `creator`, `artist`, `club`, or `bar`
4. Bot checks for duplicates/cooldowns before opening any modals:
   - **Pending** → hard block
   - **Approved** → hard block
   - **Rejected within 90 days** → block with reapply date shown
   - **Rejected 90+ days ago** → allow reapply
5. Multi-step Discord modals collect form data (2–4 modals depending on category)
6. On final submission, data is written to `partner_applications` in Supabase
7. A formatted embed card is posted to the admin Discord channel with Approve/Reject buttons
8. Admin clicks Approve → bot assigns Discord role + DMs applicant a success message
9. Admin clicks Reject → bot DMs applicant a decline message

### Bot Internals
- **In-memory session store** with 30-minute TTL — holds partial form data between modal steps. Cleaned every 5 minutes.
- **Structured logging** — every significant event logs `[ISO_TIMESTAMP] [LEVEL] message {meta}` to stdout
- **Env validation on startup** — process exits immediately if any required env var is missing
- **Race condition guard** on approve/reject — fetches current status before acting; if already reviewed, returns a warning and does nothing
- **Role assignment failure handling** — catches Discord error 50013 (bot role too low in hierarchy), alerts the admin channel with exact fix instructions
- **DM failure handling** — caught and logged; admin channel notified to follow up manually
- **Embed field truncation** — all fields capped at 1024 chars to prevent Discord API errors

## Bot Additions Required

The bot needs two additions alongside the dashboard build. Claude Code builds these too.

### 1. Dynamic Form Loading
The bot currently hardcodes all modal builders. Refactor so that:
- On startup, bot fetches `form_config` from Supabase and builds modal functions dynamically
- Config is cached in memory — never hit DB on every interaction
- Bot polls a `bot_signals` table every 60 seconds for a `reload_forms` signal
- When signal is detected, bot re-fetches `form_config`, rebuilds modals, clears the signal
- Bot also accepts a `/reload-forms` slash command (admin only, guild-restricted) that triggers the same reload immediately

The dynamic modal builder must respect all existing Discord constraints already baked into the bot:
- Max 5 fields per modal step
- `short` vs `paragraph` field types map to `TextInputStyle.Short` and `TextInputStyle.Paragraph`
- `required` boolean maps directly
- `label` and `placeholder` passed through as-is
- Modal title comes from `step_title` in `form_config`

**Critical:** The `personal` steps (step 1 and 2) apply to all categories. The bot's age validation (must be 21+, DD/MM/YYYY format) and email validation are hardcoded logic, not form config — these must remain hardcoded regardless of what's in `form_config` for the personal steps.

### 2. Bot HTTP API
Add a small Express HTTP server that runs alongside the Discord client on `BOT_API_PORT` (default 3001). The dashboard calls this for Discord-specific actions.

```
POST /api/applications/:id/approve
  Body: { reviewerName: string, note?: string }
  → Updates DB status to 'approved'
  → Assigns Discord role
  → DMs applicant (uses DM template from settings table, falls back to hardcoded default)
  → Returns: { ok: true } or { ok: false, error: string }

POST /api/applications/:id/reject
  Body: { reviewerName: string, note?: string }
  → Updates DB status to 'rejected'
  → DMs applicant
  → Returns: { ok: true } or { ok: false, error: string }

POST /api/reload-forms
  → Triggers immediate form config reload from Supabase
  → Returns: { ok: true, reloadedAt: ISO_timestamp }

GET /api/health
  → Returns: { ok: true, uptime: seconds, formsLastLoaded: ISO_timestamp }
```

**Auth:** All endpoints require `Authorization: Bearer <BOT_API_SECRET>` header. Return 401 if missing or wrong. Check server-side.

**Race condition:** The approve/reject endpoints must replicate the bot's existing guard — fetch current status before acting, return `{ ok: false, error: 'already_reviewed', currentStatus: '...' }` if not pending. The dashboard must handle this response gracefully (show a clear message, refresh the application status).

**Error handling:** Follow the bot's existing logging pattern — `[ISO_TIMESTAMP] [LEVEL] message {meta}`. Never let an unhandled error crash the HTTP server; catch everything and return structured error responses.

**Note field:** If a `note` is provided in the request body, append it to the DM: `"\n\n📝 Note from the team: [note]"`. Saves to a `review_note` column on `partner_applications` (add this column).

## New Database Tables

```sql
-- Who can access the dashboard
create table dashboard_users (
  id          uuid references auth.users primary key,
  email       text unique not null,
  name        text,
  role        text default 'member',  -- 'admin' | 'member'
  created_at  timestamptz default now(),
  created_by  uuid references dashboard_users(id),
  last_login  timestamptz
);

-- Form structure — read by bot to build Discord modals dynamically
create table form_config (
  id          bigint generated by default as identity primary key,
  category    text not null,   -- 'personal' | 'bar' | 'club' | 'artist' | 'creator'
  step        int  not null,   -- 1-indexed step number within that category
  step_title  text not null,   -- Discord modal header (max 45 chars)
  fields      jsonb not null,  -- ordered array of field objects (see shape below)
  updated_at  timestamptz default now(),
  updated_by  uuid references dashboard_users(id),
  unique(category, step)
);

-- Field object shape (element of form_config.fields jsonb array):
-- {
--   key:          string,              -- internal ID, used as customId in Discord modal and key in answers jsonb
--   label:        string,              -- shown in Discord modal (max 45 chars)
--   type:         'short'|'paragraph', -- maps to TextInputStyle.Short/Paragraph
--   required:     boolean,
--   placeholder?: string               -- max 100 chars
-- }

-- Signals from dashboard to bot
create table bot_signals (
  id          bigint generated by default as identity primary key,
  created_at  timestamptz default now(),
  signal      text not null,   -- 'reload_forms'
  processed   boolean default false,
  processed_at timestamptz
);

-- Configurable settings (role IDs, DM templates, etc.)
create table settings (
  key         text primary key,
  value       text not null,
  updated_at  timestamptz default now(),
  updated_by  uuid references dashboard_users(id)
);

-- Audit trail
create table audit_log (
  id          bigint generated by default as identity primary key,
  created_at  timestamptz default now(),
  user_id     uuid references dashboard_users(id),
  user_email  text,
  action      text,        -- 'approve' | 'reject' | 'form_edit' | 'user_invite' | 'user_remove' | 'settings_update'
  target_type text,        -- 'application' | 'form_config' | 'dashboard_user' | 'settings'
  target_id   text,
  meta        jsonb        -- old values, new values, any relevant context
);

-- Add review_note column to existing applications table
alter table partner_applications add column review_note text;
```

## Authentication & Security

### Login
Magic link only. No passwords. Flow:
1. User enters email on `/login`
2. Supabase sends magic link (expires 1 hour)
3. On click → Supabase auth session created → redirected to dashboard
4. On first login, middleware checks `dashboard_users` table — if email not found, show "Access denied. Contact Ed." and sign them out immediately.

### Roles
- `admin` — full access to everything
- `member` — applications review only (can approve/reject with notes); cannot access Forms, Team, Audit, or Settings pages

### Middleware (`middleware.ts`)
- Protect all routes under `/(dashboard)` server-side
- On every request: verify Supabase session → check `dashboard_users` → check role for admin-only routes
- No client-side-only auth gates anywhere — middleware is the single gate
- Redirect unauthenticated users to `/login`
- Redirect unauthorised role to `/` with a toast

### Security Rules
- `SUPABASE_SERVICE_ROLE_KEY` — server-side only, never in `NEXT_PUBLIC_*`, never in client components
- All Supabase writes from the dashboard go through Next.js API routes using the service role key, not the anon key
- Bot API calls from Next.js API routes only — never from client components directly
- `BOT_API_SECRET` — server-side only
- Magic links expire after 1 hour (Supabase default — document this, don't try to configure it)
- Every approve/reject/form-edit/settings-change writes to `audit_log`

## API Routes (Next.js)

All API routes are server-side only. They verify the Supabase session cookie before doing anything.

```
POST /api/applications/[id]/approve
  - Verify auth + role (member or admin)
  - Check application exists and is pending
  - Call bot API: POST [BOT_API_URL]/api/applications/[id]/approve
  - Handle already_reviewed response
  - Write audit_log
  - Return result to client

POST /api/applications/[id]/reject
  - Same pattern as approve

POST /api/applications/bulk-approve
  - Admin only
  - Array of IDs in body
  - Loop: call approve for each, collect results
  - Return { succeeded: [], failed: [] }

POST /api/applications/bulk-reject
  - Same pattern as bulk-approve

GET /api/applications/export
  - Admin or member
  - Accepts same filter params as the applications page
  - Queries Supabase, flattens answers jsonb, returns CSV

POST /api/forms/push
  - Admin only
  - Calls bot API: POST [BOT_API_URL]/api/reload-forms
  - Returns { ok: true } or error

POST /api/team/invite
  - Admin only
  - Creates dashboard_users row + sends Supabase magic link

DELETE /api/team/[id]
  - Admin only
  - Removes from dashboard_users + disables Supabase auth user

PATCH /api/settings
  - Admin only
  - Upserts key/value pairs to settings table
  - Writes audit_log
```

## Build Order

Build in this order. Each phase is shippable before the next starts.

**Phase 1 — Foundation**
- Supabase tables + RLS (all tables from schema above)
- Seed script (`seed-forms.js`)
- Auth (login page, middleware, session handling)
- Sidebar layout with nav

**Phase 2 — Form Editor**
- Form editor page (3-column layout, drag and drop, Discord preview)
- `/api/forms/push` route
- Bot: dynamic form loading + poll loop + `/api/reload-forms` endpoint

**Phase 3 — Application Review**
- Applications page (table, filters, search, export)
- Detail panel (slide-in, approve/reject with note)
- Bot: HTTP API server with approve/reject endpoints
- `/api/applications/*` routes

**Phase 4 — Admin Pages**
- Home dashboard (stats, chart, recent applications, bot status)
- Team management page
- Audit log page
- Settings page

## What NOT To Build

- Dark mode
- Mobile layout (desktop-first, Leslie uses a laptop)
- Email notifications for new applications
- Multi-server/multi-guild support
- Any public-facing pages
- Custom analytics beyond the counts on the home page
- The Discord bot from scratch — it already exists, only add the HTTP API and dynamic form loading

## Known Constraints & Gotchas

- **Discord modal limit:** Max 5 fields per step. The form editor must enforce this — disable "Add Field" at 5, show a clear tooltip explaining why.
- **Discord label limit:** 45 chars. Character counter in form editor.
- **Discord placeholder limit:** 100 chars. Character counter in form editor.
- **Discord modal title limit:** 45 chars. Character counter in form editor.
- **Bot race condition:** Approve/reject from the dashboard can conflict with someone clicking the Discord button at the same time. The bot API returns `already_reviewed` — handle this gracefully in the UI.
- **Bot DM failures:** Bot cannot DM users who have DMs closed. The bot alerts the admin Discord channel already. The dashboard should note in the UI if a DM failure was logged (add a `dm_sent` boolean column to `partner_applications`).
- **Role hierarchy:** If the bot's role isn't above partner roles in Discord, role assignment silently fails. This is a Discord config issue, not a code issue — document it in Settings page UI as a help tip.
- **Session caching in bot:** The bot's in-memory session store is lost on restart. This is intentional and acceptable — users just have to run `/apply` again. Do not attempt to persist sessions to the DB.
- **`personal` steps are shared:** The `personal` category in `form_config` applies to all application flows. The form editor should make this clear visually (label it "All Categories" and explain it in a help tooltip).
